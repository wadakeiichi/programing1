# 2027 プログラミング I — 初期設定ガイド

> **対象**: 2027 年度 プログラミング I 受講生（Mac / Windows 両対応）
> **目的**: 授業で使う開発環境（VS Code + Git + Docker Desktop + Dev Container）を構築し、全員が **同じ Linux 環境** で演習できるようにする
> **所要時間の目安**: 1〜2 時間（再起動の待ち時間を含む）

---

## 0. 全体像

授業では「**Dev Container**」という仕組みを使い、受講生全員が **同じ Linux 環境** で Python の演習を行います。そのために以下を順番に整えます。

```
あなたの PC (Mac / Windows)
   ├─ VS Code           ← コードを書くエディタ
   ├─ Git               ← 教員から教材を取得・更新する仕組み
   └─ Docker Desktop    ← Linux 環境を仮想で動かす土台
        └─ Dev Container ← VS Code が「中で」動く統一環境
              └─ Python / 演習ライブラリ（最初から揃っている）
```

**学生がやること**:

1. VS Code・Git・Docker Desktop を入れる
2. 教員リポジトリを `git clone` で取得
3. VS Code から「Dev Container で開く」
4. その中で Python コードを書いて実行する

---

## 用語の確認

| 用語 | 役割 |
|---|---|
| **VS Code** | Microsoft 製のエディタ。Python・Git・Docker と統合 |
| **Git** | バージョン管理ツール。教員リポジトリから教材を取得 |
| **Docker** | 仮想化基盤。「コンテナ」を作るソフト |
| **コンテナ** | 軽い仮想マシン。アプリと OS をまとめた箱 |
| **Dev Container** | VS Code 機能。コンテナの中を VS Code から直接編集・実行 |

> 英語の **container** は日本語で「コンテナ」とも「コンテナー」とも書きます（Microsoft 系 UI は「コンテナー」）。指すものは同じです。

# Part A: 共通セットアップ

## Step 1: 自分の PC の構成を確認する

### 🪟 Windows

1. `Windows キー + R`
2. `msinfo32` と入力 → Enter
3. 「**システムの種類**」を確認
   - `x64 ベース PC` → **x86_64** 版インストーラを選ぶ
   - `ARM ベース PC` → **ARM** 版インストーラを選ぶ

### 🍎 macOS

最近の MacBook（M1/M2/M3/M4 など）はすべて Apple Silicon（ARM）です。気にしなくて OK。Intel Mac の場合だけ x86_64 版を選びます（不明な場合は **アップルメニュー → この Mac について**で確認）。

---

## Step 2: 管理者ターミナルの開き方と権限確認 🪟

> **Windows 学生のみ。Mac 学生は Step 3 へ。**

Docker Desktop と WSL2 は **管理者権限が無いとインストールできません**。家族と共有している PC や学内貸与 PC を使う場合は、必ず最初に確認します。

### Step 2.1: 管理者ターミナル（管理者 PowerShell）を開く

これ以降の Pre-flight チェック・WSL2 セットアップ・Docker 関連の操作は、すべて **「管理者として実行」されたターミナル** から行います。**通常のターミナルでは権限不足のため失敗します**。

> 本授業では **Windows 11 のみ対応** しています。Windows 10 の人はまず Windows 11 への更新（無料）を済ませてから進めてください。

#### 基本の手順

1. 画面左下の **スタートボタン**（Windows ロゴ）を **右クリック**
2. メニューから **「ターミナル（管理者）」** を選ぶ
3. UAC（ユーザーアカウント制御）の確認画面で **「はい」** をクリック

#### 代替方法

- `Windows キー + X` でメニューが出るので、**「ターミナル（管理者）」** を選ぶ
- スタートメニューの検索ボックスに `terminal` と入力 → 検索結果の **「ターミナル」** を **右クリック** → **「管理者として実行」**

#### 開けたかの確認

ウィンドウのタイトルバーに **「管理者:」** または **「Administrator:」** と表示されていれば成功。表示されていない場合は **管理者ではない通常のターミナル** なので、一度閉じて開き直してください。

```
✓ 「管理者: Windows PowerShell」  または  「管理者: ターミナル」  ← OK
✗ 「Windows PowerShell」          または  「ターミナル」          ← NG（普通のターミナル）
```

> **以降の手順で「管理者 PowerShell で...」と書かれている箇所は、すべて Step 2.1 の方法で開いたターミナル** を指します。一度閉じてしまった場合は、毎回 Step 2.1 の手順で開き直してください。

### Step 2.2: 管理者権限の確認

開いた管理者 PowerShell に、以下を貼り付けて Enter：

```powershell
([Security.Principal.WindowsPrincipal] `
    [Security.Principal.WindowsIdentity]::GetCurrent()).IsInRole( `
    [Security.Principal.WindowsBuiltInRole] "Administrator")
```

- `True` → 次へ進む
- `False` → 自分のアカウントは管理者ではない。先に **家族／学内 IT 部門** に管理者権限化を依頼してから先に進むこと。**権限が無いまま進めると授業中に解決不可能になります**。

## Step 3: Pre-flight チェック 🪟

> **Windows 学生のみ。Mac 学生は Step 4 へ。**

「自分の PC は Docker を動かせる状態か」を **インストールする前** に一度確認します。これをやっておくと、後の Step 7 (Docker インストール) でトラブルになる確率が大幅に下がります。

**管理者 PowerShell**（Step 2.1 の方法で開く）に以下をまるごとコピペし、`Enter ↵` で実行：

```powershell
Write-Host "=== BIOS 仮想化 ===" -ForegroundColor Cyan
(Get-CimInstance Win32_Processor).VirtualizationFirmwareEnabled

Write-Host "`n=== Windows 機能 ===" -ForegroundColor Cyan
Get-WindowsOptionalFeature -Online -FeatureName `
  VirtualMachinePlatform, Microsoft-Windows-Subsystem-Linux, HypervisorPlatform |
  Select-Object FeatureName, State

Write-Host "`n=== WSL ===" -ForegroundColor Cyan
wsl --status 2>$null
wsl -l -v 2>$null

Write-Host "`n=== Hypervisor 起動設定 ===" -ForegroundColor Cyan
bcdedit /enum | Select-String hypervisorlaunchtype

Write-Host "`n=== 古い Docker の残骸 ===" -ForegroundColor Cyan
foreach ($p in 'C:\ProgramData\DockerDesktop','C:\ProgramData\Docker','C:\Program Files\Docker') {
  if (Test-Path $p) { "$p : 存在" } else { "$p : なし" }
}
```

**期待される結果**:

| 項目 | 期待値 | 違う場合 |
|---|---|---|
| `VirtualizationFirmwareEnabled` | `True` | BIOS で **Intel VT-x / AMD-V (SVM)** を有効化 → **付録 B-1** |
| `VirtualMachinePlatform` | `Enabled` | Step 7 の `wsl --install` で自動有効化される（OK） |
| `Microsoft-Windows-Subsystem-Linux` | `Enabled` | 同上 |
| `HypervisorPlatform` | `Enabled` | 同上 |
| `hypervisorlaunchtype` | `Auto` | Step 7 後に `bcdedit /set hypervisorlaunchtype auto` |
| 古い Docker 残骸 | 全て **なし** | 「存在」が出たら **付録 B-2** で削除してから次へ |

## Step 4: VS Code をインストールする 🍎🪟

1. ブラウザで <https://code.visualstudio.com/> を開く
2. 自分の OS 用のインストーラをダウンロード
   - **🍎 macOS**: 「Apple Silicon」or「Intel」を Step 1 の結果に合わせて選ぶ
   - **🪟 Windows**: x64 ベースなら **Windows**、ARM なら **User Installer (ARM64)** を選ぶ
3. ダウンロードしたファイルを実行してインストール
   - **🪟 Windows**: 「デスクトップにアイコンを作成する」にチェック推奨

### Step 4.1: VS Code の日本語化

1. VS Code を起動
2. 拡張機能パネルを開く: **`Ctrl + Shift + X`** (Win) / **`⌘ + Shift + X`** (Mac)
3. 検索ボックスに `Japanese Language Pack` と入力
4. **Japanese Language Pack for Visual Studio Code**（Microsoft 製）をインストール
5. 右下に「言語を変更して再起動」が出たらクリック
6. 再起動後、メニューが日本語になる

### Step 4.2: 必須拡張機能を入れる（host VS Code に）

VS Code の **拡張機能パネル**（`Ctrl + Shift + X` / `⌘ + Shift + X`）で検索してインストール。**似た名前の拡張機能が多い** ので、提供元（Publisher）が **Microsoft** であることを確認してから入れてください。

#### 必須

- **Dev Containers**（Microsoft 製、`ms-vscode-remote.remote-containers`）
  - Dev Container を開くための **必須** 拡張機能。これが host に無いと授業環境に入れません。

#### 推奨（任意）

- **Markdown All in One**
  - 教員配布の `.md` 教材（`docs/week3a.md` など）を読みやすく表示・編集できる
  - 入れなくても VS Code 標準の **Markdown プレビュー**（`Ctrl + Shift + V`）で読めるので、好みで判断して OK

> 📖 **配布された `.md` 教材の読み方（プレビュー表示）**
>
> 教材（`docs/week3a.md` など）は **プレビュー表示** にすると、見出し・表・絵文字が整形されて格段に読みやすくなります。
>
> 1. 左の **エクスプローラー**（いちばん上の書類アイコン）で、読みたい `.md` ファイルをクリックして開く
> 2. **プレビューを開く**:
>    - 🪟 Windows: `Ctrl + Shift + V`
>    - 🍎 macOS: `⌘ + Shift + V`
>    - または、エディタ右上の **本を分割したようなアイコン（Open Preview to the Side）** をクリックすると、左にソース・右にプレビューを **並べて** 表示できる
> 3. 元の文字（ソース）に戻したいときは、同じショートカットをもう一度押すか、タブを切り替える

> 💡 **Python・Jupyter などの開発系拡張機能は host に入れる必要はありません**: 教員リポジトリの `.devcontainer/devcontainer.json` に登録済みで、Dev Container を開いた時に **コンテナの中に自動インストール** されます。host にはあえて入れない方が、host 環境を汚さずに済みます。

### Step 4.3: VS Code の画面構成を覚える

VS Code は **4 つの主要領域** で構成されます。授業中、教員が「**サイドバーの〜**」「**ステータスバーの〜**」と言うので、名前を覚えておくとついていきやすいです。

```
+---+---------------+--------------------------------+
| A |               |                                |
| c |   Sidebar     |            Editor              |
| t |               |                                |
| i |               +--------------------------------+
| v |               |                                |
| i |               |            Panel               |
| t |               |    (Terminal / Output / ...)   |
| y |               |                                |
+---+---------------+--------------------------------+
| Status Bar  >< 開発コンテナー : ...     Ln 1, Col 1 |
+----------------------------------------------------+
  ↑        ↑                ↑                ↑
 Activity  Sidebar          Editor +         Status
 Bar       (file list)      Panel            Bar
```

| 名称 | 位置 | 何をする所か |
|---|---|---|
| **アクティビティバー** | 左端の縦帯 | エクスプローラー・検索・Git・拡張機能などのアイコン群 |
| **サイドバー** | アクティビティバーの右 | 選択したアイコンの内容を表示（ファイル一覧など） |
| **エディタ** | 中央 | コードを書く・読むメイン領域。タブで複数開ける |
| **パネル** | 下部 | **ターミナル**・出力・問題リスト・デバッグコンソール |
| **ステータスバー** | 一番下の青い帯 | Dev Container の状態（左下 `><` マーク）、行番号、Git ブランチなど |
| **コマンドパレット** | （開いた時だけ表示） | `Ctrl + Shift + P` で開く。VS Code の全機能を検索・実行 |

#### ⚠️ ターミナルが 2 種類ある — 混同に注意

VS Code を使い始めると **「ターミナル」** が **2 か所** に出てきます。**名前は同じでも、まったく別の場所で動いているもの** なので混同しないように注意。

| | 🪟🍎 **OS のターミナル** | 💻 **VS Code 内のターミナル** |
|---|---|---|
| **どこで開く** | Windows: スタートから「ターミナル（管理者）」<br>Mac: `⌘ + Space` → `terminal` | VS Code の下部パネル<br>（``Ctrl + ` `` または `Ctrl + ;`） |
| **どこで動いている** | あなたの PC 本体（Windows / macOS） | Dev Container 起動後は **コンテナの中（Linux）** |
| **使う場面** | WSL2 / Docker のインストール、PowerShell 系コマンド、Git の `clone` | 授業中の Python 実行、`make`、Linux コマンド練習 |
| **`docker -v`** | OK | Dev Container 内では普通使わない |
| **`make`** | 普通使わない | OK（教材同期はこちら） |

> **覚え方**: 「**Docker を入れる前 = OS のターミナル**」「**Dev Container に入った後 = VS Code 内のターミナル**」。
>
> 本ガイドでは Step 7 までが OS のターミナル、Step 10 以降が VS Code 内のターミナルの出番です。

# Part B: Docker Desktop のインストール

## Step 7: Docker Desktop をインストールする

### 🪟 Windows

> **重要**: **Microsoft Store からのインストールは使わない**でください。Microsoft Store 版は `C:\ProgramData\DockerDesktop` のパーミッション問題（**付録 B-3**）を非常に起こしやすく、本授業では非推奨です。

#### Step 7.1: WSL2 を先にセットアップ（重要！）

Docker Desktop より **前** に WSL2 を入れておくと、必要な Windows 機能（Virtual Machine Platform / Hypervisor Platform / WSL2 カーネル）がほぼ自動で揃い、トラブルを大幅に減らせます。

**管理者 PowerShell**（Step 2.1 の方法で開く）に以下をコピペし、`Enter ↵` で実行：

```powershell
wsl --install --no-distribution
wsl --update
wsl --set-default-version 2
```

数分かかったら、続けて以下をコピペし `Enter ↵` で **PC を再起動** します。

```powershell
shutdown /r /t 0
```

> ⚠️ **再起動は必須！**
> スリープ／サインアウト／VS Code の再起動 では不十分です。**OS ごと再起動** してください。再起動しないと Windows 機能の有効化が反映されず、後の Docker 起動で `Virtualization support not detected`（付録 B-4）が出ます。

再起動後、もう一度 **管理者 PowerShell** を開き直して、以下をコピペ＆ `Enter ↵`：

```powershell
wsl --version
wsl --status
```

`Default Version: 2` と表示されれば OK。

#### Step 7.2: Docker Desktop をインストール

1. ブラウザで <https://www.docker.com/products/docker-desktop/> を開く
2. **Download for Windows** をクリックして `Docker Desktop Installer.exe` をダウンロード
   - Step 1 で確認した CPU に合わせて x86_64 / ARM 版を選ぶ
3. ダウンロードしたファイルを **右クリック → 「管理者として実行」** で起動
   - **⚠️ ダブルクリックではなく必ず右クリック → 管理者として実行**。ダブルクリックでも一見動くが、後でパーミッション問題が出ます
4. インストール画面で **「Use WSL 2 instead of Hyper-V」にチェック** が入っていることを確認
5. 完了後、サインアウト or 再起動を促されたら従う

### 🍎 macOS

1. <https://www.docker.com/products/docker-desktop/> を開く
2. **Download for Mac** をクリック
   - Step 1 の結果に合わせて **Apple Silicon** or **Intel chip** を選ぶ
3. ダウンロードした `.dmg` を開いて Docker アプリを Applications にドラッグ
4. Docker を起動。最初に admin パスワードを求められたら入力

## Step 8: Docker Desktop を起動・自動起動の設定

### Step 8.1: 起動確認

Docker Desktop を起動すると **「Welcome to Docker」** 画面が出ます。**Skip** で OK（サインインは不要）。

メニューバー右上（🍎）または通知領域（🪟）に **クジラのアイコン 🐳** が表示され、「Docker Desktop is running」になるまで 1〜2 分待ちます。

PowerShell（Win） / ターミナル（Mac）で：

```bash
docker -v
docker ps
docker run --rm hello-world
```

最後のコマンドで `Hello from Docker!` が出れば成功です。

### Step 8.2: 自動起動の設定

ログイン時に Docker Desktop が自動で起動するようにしておくと便利です。

- **🍎 macOS**: システム設定 → 一般 → ログイン項目と機能拡張 → ログイン時に開く に Docker を追加（`+` ボタン）
- **🪟 Windows**: 設定 → アプリ → スタートアップ → **Docker Desktop Launcher** をオン

# Part C: Dev Container を起動

## Step 9: VS Code でフォルダを開く

1. VS Code を **再起動**（Docker Desktop が起動してから）
2. ファイル → **新しいウィンドウ**
3. ファイル → **フォルダーを開く**
4. **clone した `programing1` フォルダ** を選択
   - 🪟 Windows: `Documents/programing2027/programing1` （**OneDrive 配下を選ばないこと**）
   - 🍎 macOS: `~/Documents/programing2027/programing1`
   - **`programing1` の上の階層を選ばないこと**
5. 「このファイルの作成者を信頼しますか？」と出たら **「はい、作成者を信頼します」**

## Step 10: Dev Container で再度開く

1. 画面左下の **`><` マーク**（青または茶色）をクリック
   - 見えないときは VS Code ウィンドウを大きくするか、`Ctrl + Shift + P` (Win) / `⌘ + Shift + P` (Mac) でコマンドパレットを開き、`Dev Containers: Reopen in Container` と入力
2. メニューから **「コンテナーで再度開く」** または **「Reopen in Container」** を選ぶ
   - **「コンテナー構成ファイルの追加」は選ばない**。これは新規作成用で、教員提供の設定が壊れます

初回は数分（インストール待ち）かかります。完了すると左下の表示が **「開発コンテナー: Python3@desktop-linux」** のようになります。

> **重要**: 初回ビルドが完了したら、**一度 VS Code ウィンドウを閉じて、もう一度 `programing1` を VS Code で開き直してください**。これで dev container 内に正しく入った状態が保存されます。

## Step 10.1: 動作確認

VS Code 内で **`Ctrl + Shift + @`** または `Ctrl + ;` でターミナルを開き：

```bash
ls
make
```

- `ls` で `README.md`, `docs/`, `.devcontainer/` などが見える
- `make` を打つと `update_from_teacher.sh` が走り、教員リポジトリから最新の教材が取得される

これで初期設定は完了です。

# Part D: 毎回の授業での操作

## 教材を最新に更新する

VS Code 内のターミナルで毎回最初に：

```bash
make
```

これだけで教員配布のファイル（`docs/`, `.devcontainer/` など）が最新になります。学生が自分で作った `.py` ファイル等は削除されません。

詳しい挙動・注意は `docs/week3a.md` 末尾を参照。

## VS Code 必須ショートカット

| 動作 | Windows | macOS |

# 付録 B: Windows トラブル FAQ

> 今年度の授業で実際に複数の学生に発生したトラブルの予防・解決策です。

## B-1. BIOS で仮想化を有効化する

Pre-flight チェック (Step 3) で `VirtualizationFirmwareEnabled` が `False` だった場合：

1. Windows の **設定** → **システム** → **回復** → **PC の起動をカスタマイズする** → **今すぐ再起動**
2. 再起動後の青い画面で **トラブルシューティング** → **詳細オプション** → **UEFI ファームウェアの設定**
3. BIOS 画面で次の項目を探して **Enabled** に変更：
   - **Intel CPU**: `Intel Virtualization Technology` / `Intel VT-x` / `VT-d`
   - **AMD CPU**: `SVM Mode` / `AMD-V`
4. **設定を保存して PC を再起動**（多くの場合 F10 → Yes）
5. Pre-flight チェック (Step 3) をもう一度実行して `True` になっていることを確認

> ⚠️ **再起動を忘れない！** BIOS で「Save & Exit」を選んだだけで Windows が起動しないとき、もう一度同じ手順で BIOS に入って設定が反映されているか確認してください。

## B-2. 古い Docker の残骸を削除する

Pre-flight チェックで「存在」が出た場合、または Docker Desktop インストール時に下記 B-3 のエラーが出る場合の対処です。

**管理者 PowerShell**（Step 2.1 の方法で開く）に以下を **すべて選択 → コピー → ターミナルに貼り付け → `Enter ↵`** で実行：

```powershell
$path = "C:\ProgramData\DockerDesktop"

$admins = (New-Object System.Security.Principal.SecurityIdentifier `
    "S-1-5-32-544").Translate([System.Security.Principal.NTAccount]).Value
$system = (New-Object System.Security.Principal.SecurityIdentifier `
    "S-1-5-18").Translate([System.Security.Principal.NTAccount]).Value

Stop-Service -Name com.docker.service -ErrorAction SilentlyContinue
Stop-Process -Name "Docker Desktop" -Force -ErrorAction SilentlyContinue

if (Test-Path $path) {
    takeown /F $path /A /R /D Y
    icacls $path /grant "${admins}:(OI)(CI)(F)" /T
    Remove-Item $path -Recurse -Force
}

New-Item -ItemType Directory -Path $path -Force
icacls $path /setowner $admins
icacls $path /grant "${admins}:(OI)(CI)(F)" "${system}:(OI)(CI)(F)" /inheritance:e

(Get-Acl $path).Owner   # BUILTIN\Administrators と表示されれば OK
```

> ⚠️ **再起動が必要！** その後 PC を再起動（`shutdown /r /t 0` または通常の再起動）して、Step 7.2 から再開してください。

## B-3. `For security reasons C:\ProgramData\DockerDesktop must be owned by an elevated account`

Docker Desktop のインストール時に `C:\ProgramData\DockerDesktop` フォルダの所有者またはアクセス権が適切でないため、インストーラが安全上の理由で処理を中断している状態です。**典型的には、以前のインストール失敗や手動削除の途中で、フォルダが通常ユーザー所有のまま残ったり権限設定が壊れたりした場合に起こります**。

**主な原因**:

- `C:\ProgramData\DockerDesktop` が以前の Docker の残骸として残っている
- そのフォルダの所有者が管理者グループではなく通常ユーザーになっている
- アクセス権が壊れていて、インストーラが安全に利用できないと判断している
- Microsoft Store 経由のインストールで、既存フォルダの権限問題を引き継いでいる

**対処手順**:

1. Docker Desktop のインストール画面と Microsoft Store を閉じる
2. **B-2** の PowerShell スクリプトを **管理者 PowerShell** で実行（フォルダ所有者を Administrators に戻し、権限を再付与）
3. PC を **再起動**
4. **公式サイト**（<https://www.docker.com/products/docker-desktop/>）から `Docker Desktop Installer.exe` をダウンロードし、**右クリック → 「管理者として実行」** で再インストール

**確認方法**: 次のコマンドで所有者を確認できます。

```powershell
(Get-Acl "C:\ProgramData\DockerDesktop").Owner
```

`BUILTIN\Administrators` もしくはそれに相当する管理者グループになっていれば、状態は正常です。

> **補足**:
> - 画面上の `¥` は、日本語環境での `\` 表示です（同じ文字を指します）
> - インストール時は管理者権限が必要です
> - 再インストール前に Windows の再起動を挟むと、ロックされたファイルの問題を避けやすくなります

## B-4. `Virtualization support not detected`（タスクマネージャーでは仮想化「有効」なのに）

**症状**: Docker Desktop 起動時に `Virtualization support not detected` と表示されて起動できない。BIOS / UEFI 上では仮想化は有効になっているのに、Docker からは仮想化を使えない状態。

**問題の本体**: BIOS 側ではなく、**Windows 側の WSL2 / Virtual Machine Platform / Hypervisor の設定**である可能性が高い。

**考えられる主な原因**:

- **WSL2** が未導入、または壊れている
- **Virtual Machine Platform** が無効
- **Hypervisor** の起動設定が無効になっている
- Docker Desktop 用の内部 WSL 環境（`docker-desktop` など）が破損している

**対処の優先順位**:

1. WSL2 を導入・更新する
2. Virtual Machine Platform を有効にする
3. Hypervisor の起動を `auto` にする
4. **OS ごと再起動**する
5. それでもだめなら `docker-desktop` を再生成する

**対処**: **管理者 PowerShell**（Step 2.1 の方法で開く）に以下をコピペし、`Enter ↵` で実行：

```powershell
wsl --install --no-distribution
wsl --update
wsl --set-default-version 2

dism.exe /online /enable-feature /featurename:Microsoft-Windows-Subsystem-Linux /all /norestart
dism.exe /online /enable-feature /featurename:VirtualMachinePlatform /all /norestart
dism.exe /online /enable-feature /featurename:HypervisorPlatform /all /norestart

bcdedit /set hypervisorlaunchtype auto
```

> ⚠️ **再起動が必須！** スリープ／サインアウトでは不十分。続けて以下を実行して **OS ごと再起動** してください。

```powershell
shutdown /r /t 0
```

再起動後、もう一度 **管理者 PowerShell** を開き、以下をコピペ＆ `Enter ↵` で確認：

```powershell
wsl --status
bcdedit /enum | findstr /i hypervisor
```

`hypervisorlaunchtype: Auto` が表示されれば OK。Docker Desktop を起動し直す。

それでも同じエラーが出る場合は、Docker Desktop 用の WSL 仮想ディスクが壊れている可能性があるので、**最終手段**として、**管理者 PowerShell** で以下をコピペ＆ `Enter ↵`：

```powershell
wsl --shutdown
wsl --unregister docker-desktop
wsl --unregister docker-desktop-data
```

> **注意**: Docker 内に作ったコンテナ／イメージが消えます。Programing-I 初期段階なら問題なし。

### B-4 補足: 原因切り分け用の確認コマンド

「どこで詰まっているのか」をさらに切り分けたい場合、**管理者 PowerShell** で以下を実行して結果を見比べる：

```powershell
systeminfo
wsl --status
wsl --version
wsl -l -v
bcdedit /enum | findstr /i hypervisor
```

特に重要なのは、最後のコマンドの結果に以下が含まれていること：

```
hypervisorlaunchtype    Auto
```

ここが `Off` になっていると、**BIOS で仮想化が有効でも Docker Desktop からは仮想化を使えない状態**になります。`Off` の場合は上の `bcdedit /set hypervisorlaunchtype auto` を再度実行 → 再起動。

## B-5. Docker Desktop を完全リセットする（最終手段）

何度試しても動かないとき。**管理者 PowerShell**（Step 2.1 の方法で開く）に以下をコピペし、`Enter ↵` で実行：

```powershell
# Docker Desktop 終了
Stop-Process -Name "Docker Desktop" -Force -ErrorAction SilentlyContinue
Stop-Service -Name com.docker.service -ErrorAction SilentlyContinue

# 全削除（既存の Docker データも消えます）
Remove-Item "C:\ProgramData\Docker"        -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item "C:\ProgramData\DockerDesktop" -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item "C:\Program Files\Docker"      -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item "$env:LOCALAPPDATA\Docker"     -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item "$env:APPDATA\Docker"          -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item "$env:APPDATA\Docker Desktop"  -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item "$env:USERPROFILE\.docker"     -Recurse -Force -ErrorAction SilentlyContinue
```

> ⚠️ **再起動が必要！** その後 PC を再起動（`shutdown /r /t 0`）してから Step 7.1（WSL2）から順にやり直してください。

# 付録 C: macOS トラブル FAQ

## C-1. `Cannot connect to the Docker daemon`

**原因**: Docker Desktop アプリ本体が起動していない／起動中。

**対処**: Launchpad から Docker を起動。メニューバー右上のクジラのアイコンが「running」になるまで 1 分ほど待ってから再試行。

## C-2. Apple Silicon Mac (M1/M2/M3/M4) で「Rosetta が必要」

**原因**: Apple Silicon Mac で Intel バイナリを必要とするコンテナを動かそうとした。

**対処**: ダイアログに従って Rosetta をインストール。または Docker Desktop 設定 → General → **「Use Rosetta for x86_64/amd64 emulation」** を有効化。

## C-3. ディスク容量不足で Docker が動かない

**原因**: Docker Desktop はデフォルトで数十 GB を使う。

**対処**: 不要なイメージを削除。VS Code 内ターミナルで：

```bash
docker system prune -a
```

それでも足りない場合は Docker Desktop 設定 → Resources → Disk image size を確認。
